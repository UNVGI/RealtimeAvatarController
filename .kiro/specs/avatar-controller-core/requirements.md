# 要求定義書: Realtime Avatar Controller Core

## 1. 概要

### 1.1 目的
本ドキュメントは、VTuberシステム向けのアバター制御方法を切り替える汎用ツール「Realtime Avatar Controller」の初期バージョン（Core）に対する要求を定義するものである。

### 1.2 スコープ
本ツールは、単独での運用を主目的とするのではなく、VTuberシステムの一コンポーネントとして組み込まれることを前提とする。モーションキャプチャ（MoCap）入力からアバターへのモーション反映までの制御レイヤーを提供する。

### 1.3 用語定義
| 用語 | 定義 |
|------|------|
| Slot | 1人のMoCapアクターに対して行われる一連の設定セット（アバター、MoCapソース、表情制御、リップシンクデバイス等） |
| MoCapアクター | 物理的にMoCap機材を装着して演技を行う人物 |
| MoCapソース | モーションデータの供給元（例: VMCプロトコル経由のデバイス／アプリケーション） |
| アバター | Unity上で制御対象となる3Dキャラクタ |
| Weight | キャッシュされたモーションデータをアバターに適用する際の重み値（0.0〜1.0） |
| Humanoid | Unity標準のHumanoid Avatar形式 |
| Generic | Unity標準のGeneric Avatar形式 |

### 1.4 開発初期段階における明示的スコープ
| カテゴリ | 初期段階で対応 | 将来対応（受け口のみ用意） |
|----------|----------------|----------------------------|
| アバター供給 | プロジェクトビルトイン | Addressable Asset System |
| MoCapソース | VMC（バーチャルモーションキャプチャ） | その他のMoCapツール群 |
| モーション形式 | Humanoid | Generic |
| 表情制御 | インターフェースのみ | 実装 |
| リップシンク | インターフェースのみ | 実装 |

## 2. プロジェクト基盤要求

### 要求 2.1: Unityバージョン
**ユーザーストーリー:** 開発者として、最新のUnityランタイム機能を活用したいので、ツールは Unity 6000.3.10f1 をターゲットバージョンとして動作すること。

**受入基準:**
- WHEN プロジェクトを開く THEN Unity 6000.3.10f1 で警告なく起動できる
- THE `ProjectVersion.txt` は `6000.3.10f1` を示す
- THE パッケージの `package.json` における `unity` フィールドは `6000.3` 以降を要求する

### 要求 2.2: リポジトリ構成
**ユーザーストーリー:** 開発者として、プロジェクト管理を明確化したいので、Unityプロジェクトはリポジトリ直下の所定フォルダに集約する。

**受入基準:**
- THE Unityプロジェクト一式は `RealtimeAvatarController/` フォルダ内に配置される
- THE リポジトリルートには `RealtimeAvatarController/` 以外にUnityプロジェクトが存在しない

### 要求 2.3: UPM配布可能性
**ユーザーストーリー:** 他プロジェクトの開発者として、本ツールを容易に導入したいので、UPM（Unity Package Manager）経由で導入可能な形で提供されること。

**受入基準:**
- THE ツール本体は `package.json` を含むUPM準拠ディレクトリ構成を持つ
- WHEN 他プロジェクトから Git URL もしくは local パス経由で参照 THEN 追加手順無しで利用可能となる
- THE サンプルUI／サンプルシーン等は UPM の `Samples~` 機構を用いて提供される
- THE ツール本体（機能部）と UI サンプルは別ディレクトリ／別スコープで管理される

### 要求 2.4: 機能部とUI部の分離
**ユーザーストーリー:** ツール利用者として、自前のUIに組み込みたいので、機能部のAPIのみで制御可能であり、UIに依存しないこと。

**受入基準:**
- THE 機能部アセンブリは UI 層（UGUI／UI Toolkit 等）に依存しない
- THE UI はサンプルアセットとして提供され、機能部のパブリックAPIのみを利用する
- WHEN UIサンプルを導入せずに機能部のみを利用 THEN コンパイルおよびランタイム動作が成立する

## 3. Slot システム要求

### 要求 3.1: Slot 概念
**ユーザーストーリー:** VTuberシステム運用者として、複数のアクターを同時に扱いたいので、1アクター分の設定セットを「Slot」として管理したい。

**受入基準:**
- THE 各 Slot は以下の設定項目を個別に保持する:
  - 使用アバター
  - MoCapソース（入力）
  - 表情操作手段（受け口のみ）
  - リップシンク音声入力デバイス（受け口のみ）
  - モーション適用 Weight 値
- THE 各 Slot の設定は他 Slot の設定と独立して変更可能である

### 要求 3.2: Slot の動的増減
**ユーザーストーリー:** 運用者として、配信シーンに応じてアクター数を変えたいので、ランタイム中であっても Slot を追加・削除できること。

**受入基準:**
- WHEN API 経由で Slot 追加を要求 THEN 新規 Slot が生成され識別子が返却される
- WHEN API 経由で Slot 削除を要求 THEN 該当 Slot に紐づくリソースが解放される
- WHEN Slot が削除される THEN 該当 Slot が制御していたアバターへのモーション流し込みは停止する
- THE Slot の最大数はツール側で固定上限を設けない（実質的にはメモリ／性能制約に依存）

### 要求 3.3: Slot 設定の永続化余地
**ユーザーストーリー:** 運用者として、設定を再利用したいので、Slot 設定はシリアライズ／デシリアライズ可能な形式で保持されること。

**受入基準:**
- THE Slot 設定は Unity シリアライズ可能な型で構成される
- THE Slot 設定は ScriptableObject 等を用いてアセット化できる構造を持つ

## 4. アバター供給要求

### 要求 4.1: プロジェクトビルトイン供給（初期対応）
**ユーザーストーリー:** 開発者として、検証用アバターを手早く利用したいので、プロジェクトに事前組み込みされたアバターを Slot に割り当てられること。

**受入基準:**
- WHEN プロジェクト内に配置された Prefab 形式のアバターを指定 THEN Slot に割り当て可能である
- THE Slot に割り当てられたアバターは Scene 上にインスタンス化され、ツール管理下に置かれる

### 要求 4.2: Addressable Asset System 対応の拡張余地（将来）
**ユーザーストーリー:** 将来の運用者として、アバターを動的追加したいので、Addressable 経由でアバターを供給できる設計余地を持っていること。

**受入基準:**
- THE アバター供給口は「供給元（Provider）」として抽象化され、具象実装を差し替え可能である
- THE ビルトイン供給 Provider と Addressable 供給 Provider が同一の抽象に従うことが設計上保証される
- 注記: 初期段階で Addressable Provider は実装しない。抽象のみ用意する。

---

## 5. MoCap ソース要求

### 要求 5.1: MoCap ソースの抽象化
**ユーザーストーリー:** 拡張者として、新規 MoCap ツールに対応したいので、MoCap ソースは抽象レイヤーを介して追加可能であること。

**受入基準:**
- THE MoCap ソースは抽象インターフェース（例: `IMoCapSource`）として定義される
- WHEN 新規 MoCap ツール対応を追加 THEN 抽象インターフェース実装クラスの追加のみで対応可能である
- THE 抽象インターフェースは、今後追加されうる MoCap ツール（任意の通信プロトコル、任意のデータ構造）を吸収できる拡張性を持つ

### 要求 5.2: VMC 対応（初期対応）
**ユーザーストーリー:** VTuber として、VMC プロトコル対応機材を使いたいので、初期バージョンで VMC ソースが利用可能であること。

**受入基準:**
- THE `IMoCapSource` 実装として VMC プロトコル対応実装が同梱される
- WHEN VMC 対応デバイス／アプリから OSC データを受信 THEN モーションデータとして内部表現に変換される
- THE VMC 受信ポート等の通信パラメータは Slot 単位で個別指定可能である

### 要求 5.3: Slot と MoCap ソースの組み合わせ
**ユーザーストーリー:** 運用者として、アクターごとに異なる MoCap 機材を使いたいので、Slot 単位で MoCap ソースを選択可能であること。

**受入基準:**
- THE 各 Slot は 1 つの MoCap ソースインスタンスを紐づけて保持する
- WHEN Slot の MoCap ソースを別種類に変更 THEN モーションの流し込みが新ソースに切り替わる

## 6. モーション形式要求

### 要求 6.1: Humanoid 対応（初期対応）
**ユーザーストーリー:** 利用者として、汎用的なキャラクタモデルを扱いたいので、Unity Humanoid 形式のアバターで動作すること。

**受入基準:**
- WHEN Humanoid 設定されたアバターを Slot に割り当てた THEN MoCap ソースからのモーションが反映される
- THE 内部モーションキャッシュは Humanoid 骨格（Muscle ／ HumanPose 等）を取り扱える表現を持つ

### 要求 6.2: Generic 対応の拡張余地（将来）
**ユーザーストーリー:** 将来の拡張者として、任意骨格モデルを扱いたいので、Generic 形式への拡張余地があること。

**受入基準:**
- THE モーション形式は抽象化され、Humanoid／Generic を同一の枠組みで切り替え可能である
- 注記: 初期段階で Generic 実装は行わず、抽象構造のみ用意する。

---

## 7. モーション適用パイプライン要求

### 要求 7.1: 内部キャッシュ方式
**ユーザーストーリー:** システム設計者として、ランタイム中のシームレスな切替および応用利用を実現したいので、受信したモーションデータを直接アバターに書き込まず、内部キャッシュを介すこと。

**受入基準:**
- WHEN MoCap ソースからモーションデータを受信 THEN 該当 Slot の内部キャッシュに書き込まれる
- THE 内部キャッシュはアバター固有の骨格に依存しない中立表現を保持する
- THE アバターへの反映は内部キャッシュからの読出しに基づき行われる

### 要求 7.2: Weight による適用
**ユーザーストーリー:** 運用者として、モーションの適用度合いを動的に調整したいので、Weight 値に従ってアバターへ流し込まれること。

**受入基準:**
- THE 各 Slot は 0.0〜1.0 の Weight 値を保持する
- WHEN Weight が 0.0 THEN 該当 Slot のモーションはアバターに反映されない
- WHEN Weight が 1.0 THEN 該当 Slot のモーションが完全にアバターに反映される
- WHEN Weight が中間値 THEN モーションは指定度合いでアバターに適用される
- THE Weight はランタイム中に動的変更可能である

### 要求 7.3: ランタイム切替のシームレス性
**ユーザーストーリー:** 配信者として、配信中に MoCap 機材を切り替えたいので、切替時にアバターが破綻しないこと。

**受入基準:**
- WHEN 稼働中の Slot の MoCap ソースを別ソースに差し替え THEN アバターは継続的にポーズを保持する
- WHEN 稼働中の Slot のアバターを別アバターに差し替え THEN 新アバターに現在のキャッシュ内容が適用される

### 要求 7.4: 1アクター多アバター／多アクター1アバターの応用余地
**ユーザーストーリー:** 応用利用者として、1人のアクターが複数アバターを同時に動かしたいので、Slot とアバターは柔軟な対応関係を取れること。

**受入基準:**
- THE 複数 Slot が同一アバターを参照することは仕様上禁止されない
- THE 複数 Slot がそれぞれ別アバターを同時に制御することは仕様上禁止されない
- THE 1 Slot 内のキャッシュは他 Slot によって上書きされない

## 8. 表情制御要求（受け口のみ）

### 要求 8.1: 表情制御抽象インターフェース
**ユーザーストーリー:** 将来の拡張者として、任意の表情制御手段を組み込みたいので、Slot に表情制御手段を紐づけるインターフェースが用意されていること。

**受入基準:**
- THE 表情制御は抽象インターフェース（例: `IFacialController`）として定義される
- THE 各 Slot は 1 つの表情制御手段を紐づけて保持できる
- 注記: 初期段階で具体実装は行わない。

---

## 9. リップシンク要求（受け口のみ）

### 要求 9.1: リップシンク入力デバイス抽象
**ユーザーストーリー:** 将来の拡張者として、各種音声入力デバイスからリップシンクを駆動したいので、入力デバイス指定の受け口が用意されていること。

**受入基準:**
- THE リップシンクは抽象インターフェース（例: `ILipSyncSource`）として定義される
- THE 各 Slot はリップシンク音声入力デバイスを個別設定可能である
- 注記: 初期段階で具体実装は行わない。

---

## 10. 非機能要求

### 要求 10.1: 拡張性
- THE MoCapソース／アバター供給／表情制御／リップシンク の各カテゴリは、将来の実装追加に対し既存コードの改変を最小化する設計とする（Open-Closed Principle）。

### 要求 10.2: 性能
- THE Slot 複数稼働時にも、主要 MoCap ソース（VMC）の受信レート（一般的に 30〜90 Hz 程度）に追従可能である。
- THE モーションキャッシュおよび適用処理は、Unity のメインスレッドおよび Update / LateUpdate のタイミング制約を考慮した設計とする。

### 要求 10.3: 依存
- THE 機能部コアは UGUI／UI Toolkit 等の UI フレームワークに依存しない。
- THE 機能部コアは Unity 標準パッケージおよび VMC 実装に必要な最小依存以外に追加パッケージを要求しない。

### 要求 10.4: 保守性
- THE 抽象インターフェースと具象実装は別アセンブリまたは別名前空間で整理される。
- THE 将来対応予定の抽象（Addressable Provider、Generic モーション、表情、リップシンク）は、未実装であることが API ドキュメントまたは XMLDoc で明示される。
